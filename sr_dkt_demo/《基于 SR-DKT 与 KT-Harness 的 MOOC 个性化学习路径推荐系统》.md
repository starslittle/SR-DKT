## 1. 选题定位

推荐题目：

**《基于 SR-DKT 双状态与学习动作推荐的 MOOC 个性化学习路径推荐系统》**

这个方向从原来的“作文辅导系统”转向“MOOC 学习行为分析与个性化路径推荐”。相比作文方向，它更适合作为硕士论文选题，原因是数据更容易获取、指标更客观、实验更容易离线完成，也更容易做出可量化的模型对比和消融实验。

核心思路是：利用 MOOC 平台中的学习行为日志，先通过储存-提取双状态认知行为知识追踪模型 SR-DKT（Storage-Retrieval Deep Knowledge Tracing）估计学习者在各知识点上的长期储存强度和短期提取强度，再把双状态结果映射为学习动作推荐策略，输出“观看讲解、低难度练习、主动检索测验、间隔复习、讨论复述、进阶挑战”等认知动作，最后再映射到具体 MOOC 资源，并通过离线反事实验证不断优化策略。

## 3. 核心研究问题

本文可以围绕一个核心问题展开：

**如何基于学习者的历史行为数据和双状态知识表征，动态生成适合其当前认知状态的 MOOC 个性化学习动作与学习路径？**

这一问题对应 MOOC 平台长期存在的“高注册、低完成”困境，其根源可以概括为三个缺失：

```text
缺失1：不知道学生“哪里不会”
→ 知识状态感知粗糙

缺失2：不知道学生“什么时候会忘”
→ 遗忘建模缺失

缺失3：推荐了但效果好不好没人管
	→ 推荐策略不评估、不迭代
```

可以进一步拆成三个子问题：

1. 如何从 MOOC 行为日志中建模学习者的储存强度与提取强度两个认知状态？
2. 如何显式建模主动检索行为与被动接收行为在长期储存和短期提取上的差异？
3. 如何将 SR-DKT 输出的双状态表征转化为学习动作推荐信号，并通过反事实离线验证机制实现推荐质量的持续优化？

## 4. 总体技术路线

整体流程可以设计为：

```text
步骤1：下载并清洗 MOOC 公开数据集
        ↓
步骤2：构建学习者行为序列与知识点序列
        ↓
步骤3：训练 SR-DKT 双状态知识追踪模型
        ↓
步骤4：设计学习动作推荐与离线验证框架
        ↓
步骤5：通过双状态→动作映射生成个性化学习动作与复习计划
        ↓
步骤6：与传统 DKT、DKT-F、BEKT、协同过滤、随机推荐、无知识追踪 Agent 等基线对比
        ↓
步骤7：通过主实验、消融实验、反事实验证和案例分析验证效果
```

这个技术路线的重点不是简单做一个推荐系统，而是形成“SR-DKT 双状态感知 + 动作推荐 + 反事实验证 + Correct 纠正”的教育闭环。

### 5.1 系统总体架构

```text

║      双状态驱动的动作推荐与教育验证系统            ║
╠═══════════════════════════════════════════════════╣
║                                                   ║
║  INFORM 层：告知                                  ║
║  感知 Agent：整理原始 MOOC 行为序列                ║

║                       ↓                           ║

║  诊断 Agent：识别薄弱知识点、退课风险、遗忘风险     ║
║  将推荐候选空间约束在当前薄弱知识点及其相关资源     ║
║                       ↓                           ║
║  GENERATE 层：生成                                ║
║  动作推荐 Agent：生成个性化学习动作                ║
║  调度 Agent：基于 SM-2 生成复习时间表              ║
║                       ↓                           ║
║  VERIFY 层：验证                                  ║
║  反思 Agent：计算推荐前后掌握度提升 ΔKS            ║
║                       ↓                           ║
║  CORRECT 层：纠正                                 ║
║  若 ΔKS 低于阈值，则反馈动作推荐 Agent / 调度 Agent ║

║                                                   ║
╚═══════════════════════════════════════════════════╝
```

从 Harness Engineering 视角看，本文系统的关键不是“堆叠多个 Agent”，而是把 SR-DKT 输出的双状态向量转化为 Agent 行为的动态护栏和验证标准。传统 Harness 多用于软件工程场景，约束多来自代码规范、架构规则和测试用例；本文将 Harness 思想迁移到 MOOC 个性化学习场景，使约束、验证和纠正都由学习者实时双状态驱动。

### 5.2 各智能体职责

| Agent | 职责 | 输入 | 输出 |
|---|---|---|---|
| 感知 Agent | Inform：行为日志解析 | 原始 MOOC 日志 | 结构化行为序列 |
| 知识追踪 Agent | Inform：双状态估计 | 行为序列、知识点序列 | 双状态向量 `S_t, R_t` |
| 诊断 Agent | Constrain：风险识别与候选空间约束 | 掌握度向量、时间特征 | 薄弱知识点、候选资源约束 |
| 推荐 Agent | Generate：学习动作生成 | 双状态向量、诊断结果、课程知识图谱、约束候选集 | 下一步学习动作 |
| 调度 Agent | Generate：复习安排 | 掌握度变化、遗忘概率 | 复习时间、复习内容 |
| 反思 Agent | Verify / Correct：动作验证与策略纠正 | 推荐后学习效果、`ΔKS` | 策略调整参数 |

## 6. 核心模块设计

### 6.1 SR-DKT 双状态知识追踪模块

#### 6.1.1 理论基础：记忆双强度理论与测试效应

SR-DKT（Storage-Retrieval Deep Knowledge Tracing）的核心理论依据来自 Bjork 的 **记忆双强度理论（New Theory of Disuse）** 与测试效应：记忆不是单一强度，而是由长期储存强度（Storage Strength）和短期提取强度（Retrieval Strength）共同表征。主动检索、测验和自我测试更能提升长期储存，而被动观看和阅读更多影响短期提取状态。

因此，本文不再把学习状态压缩为一个单一掌握度，而是将其解耦为两个状态：

| 状态 | 行为类型 | 认知机制 | 遗忘特点 |
|---|---|---|---|
| Storage 强度 `S_t` | 做题、测验；论坛发帖、提问、解释概念作为可扩展主动行为 | 长期储存与知识固化 | 变化慢，遗忘较慢 |
| Retrieval 强度 `R_t` | 看视频、浏览评论、阅读材料、观看回放 | 短期提取与当前可回忆性 | 变化快，遗忘较快 |

这一划分使模型设计具备明确的理论来源：**“能做对”不等于“记得住”，短期提取和长期储存应该分别建模**。

#### 6.1.2 传统 DKT 的问题

传统 DKT 使用 RNN / LSTM 根据学生历史答题序列预测下一次答题表现：

```text
传统 DKT：
(习题A, 对) → (习题B, 错) → (习题C, 对)
        ↓
同一个 LSTM
        ↓
预测下一题答对概率或知识点掌握度
```

但在 MOOC 场景中，学生行为更复杂：

```text
MOOC 行为序列：
看完视频《递归基础》        （被动接收）
做题“汉诺塔”答对            （主动检索）
回放视频片段                （被动接收）
论坛发帖“递归和迭代的区别”  （主动检索，可扩展行为，当前核心实验不纳入）
```

如果仍然使用单一 LSTM 处理上述行为，就会把“看视频”和“做题答对”对知识状态的贡献混在一起，容易高估被动学习的效果，也难以刻画不同学习行为对应的遗忘速度。

#### 6.1.3 SR-DKT 总体结构

SR-DKT 的基本思想是：不再把 MOOC 学习状态压缩成单一掌握度，而是分别建模长期储存强度和短期提取强度，再通过双状态融合头预测下一次可提取性与长期掌握水平。

这里仍然保留主动 / 被动两条行为路径，但论文方法名称强调的是 **双状态（Storage-Retrieval）**，以避免与 DDKT 中“主观难度 / 客观难度”的双通道概念混淆。

```text
MOOC 原始行为序列
     ↓ 行为分类
主动行为 → LSTM_active（更新 S_t）→ Storage Head → S_t
被动行为 → LSTM_passive（更新 R_t）→ Retrieval Head → R_t
                    ↓
            [双状态融合层]
            P(correct)=σ(w1·S_t + w2·R_t - b)
                    ↓
            双状态输出向量
```

其中，`λ_S` 表示长期储存遗忘速率，`λ_R` 表示短期提取遗忘速率。根据记忆双强度理论，本文设置并学习满足如下约束的差异化遗忘机制：

```text
λ_R > λ_S
```

也就是说，短期提取状态衰减更快，长期储存状态保持更稳定。

#### 6.1.4 行为分类与输入特征

SR-DKT 的前处理阶段需要把原始 MOOC 日志转换为结构化学习事件，并标记其对 Storage 与 Retrieval 的影响方向。

```python
def classify_behavior(event):
    if event["type"] in ["problem", "quiz"]:
        return "storage_boost"
    if event["type"] in ["video", "material_read", "browse", "review"]:
        return "retrieval_boost"
    return "unknown"
```

Storage 分支输入特征可以设计为：

```text
[concept_id_embedding,
 correct,
 attempt_count,
 Δt]
```

Retrieval 分支输入特征可以设计为：

```text
[concept_id_embedding,
 watch_ratio,
 is_replay,
 Δt]
```

基于当前 MOOCCubeX 实验实现，本文将 `user-problem.json` 中的答题记录作为 Storage 分支的主要输入，将 `user-video.json` 中提取的观看比例 `watch_ratio` 与回放标记 `is_replay` 作为 Retrieval 分支的主要输入。`comment.json` / `reply.json` 中的论坛发帖、回复等行为可作为 Storage 分支的扩展输入，但不纳入本文当前核心实验。

#### 6.1.5 双状态差异化遗忘机制

对于 Storage 状态，本文使用较小遗忘速率建模长期储存：

```text
S_t^{decay} = S_{t-1} ⊙ exp(-λ_S · Δt)
```

对于 Retrieval 状态，本文使用较大遗忘速率建模短期提取的快速衰减：

```text
R_t^{decay} = R_{t-1} ⊙ exp(-λ_R · Δt)
```

其中，`Δt` 表示相邻学习事件之间的时间间隔，`⊙` 表示逐元素乘法。

需要强调两点实现细节，二者直接关系到差异化遗忘是否真正生效：

1. **衰减作用在进入预测的隐状态上，而非仅作用于标量强度。** 本文将遗忘衰减施加在两条 LSTM 输出的隐状态 `h_S`、`h_R` 上（`h_S^{decay} = h_S ⊙ exp(-λ_S·Δt)`，Retrieval 同理），随后这两个**衰减后的隐状态**直接参与融合与预测。这样 `λ_S`、`λ_R` 会经由交叉熵损失获得真实梯度，差异化遗忘对最终预测产生实质影响，而不是退化为一个仅调节融合权重的旁路。

2. **`λ_S` 与 `λ_R` 为两个相互独立的可学习参数，`λ_R > λ_S` 由软约束惩罚项鼓励，而非由参数化恒等式强制写死。** 具体地，二者均通过 `softplus` 保证正值并各自自由学习，训练损失中加入惩罚项 `L_λ = ReLU(λ_S − λ_R + margin)` 以鼓励 `λ_R > λ_S`。因此本文报告的 `λ_R > λ_S` 是模型在数据上**学习得到**的结果，而非结构上必然成立的恒等式——这一点保证了"差异化遗忘"作为实验发现的可信度。

直观上：

```text
学生做题做对“递归”：
→ Storage 显著提升，长期保持更稳

学生只是看完“递归”视频：
→ Retrieval 提升明显，但遗忘快
```

#### 6.1.6 双 LSTM 独立更新

主动行为到来时，更新 Storage 分支：

```text
S_t = LSTM_S(x_t^{storage}, S_t^{decay})
```

被动行为到来时，更新 Retrieval 分支：

```text
R_t = LSTM_R(x_t^{retrieval}, R_t^{decay})
```

两条 LSTM 不共享参数。这样可以避免把“长期储存规律”和“当前可提取规律”强行压缩到同一个隐状态中，也能让模型分别学习两类状态对后续作答的不同影响。

#### 6.1.7 注意力融合层

双状态并不意味着固定认为 Storage 永远更重要。不同学生、不同阶段、不同知识点下，长期储存和短期提取对最终作答表现的贡献可能动态变化。因此，本文引入双状态融合层：

```text
α = softmax(W_α · [S_t ; R_t] + b_α)

h_final = α_S · h_S^{decay} + α_R · h_R^{decay}
```

其中 `S_t`、`R_t` 是从衰减后隐状态导出的可解释双状态强度（用于注意力门控与可视化解释），而真正参与加权融合的是**衰减后的隐状态** `h_S^{decay}`、`h_R^{decay}`。融合权重 `α` 由双状态强度动态生成：当学生近期主要进行习题练习时，模型可以提高 Storage 权重；当学生刚开始接触新知识、Retrieval 更容易波动时，模型可以提高 Retrieval 权重。融合层使 SR-DKT 不只是机械相加，而是根据认知状态动态整合双状态信息。

#### 6.1.8 输出与训练目标

最终输出为学生在时间 `t` 对各知识点的双状态向量：

```text
K_s,t = [(S_1, R_1), (S_2, R_2), ..., (S_n, R_n)]
```

其中每个 `S_i` 表示长期储存强度，每个 `R_i` 表示短期提取强度。

若训练任务为预测学生下一次答题是否正确，则输出层为：

```text
ŷ_t = sigmoid(W_o · h_final + b_o)
```

训练损失使用二分类交叉熵：

```text
L = -Σ_t [y_t log(ŷ_t) + (1 - y_t) log(1 - ŷ_t)]
```

#### 6.1.9 运行示例

以“递归”知识点为例：

```text
Day 1 09:00：看视频《递归基础》，观看 85%，无回放
→ Retrieval 更新：R_rec 0.00 → 0.35

Day 1 09:30：做题“斐波那契数列”，答错，用时 240 秒
→ Storage 更新：S_rec 0.00 → 0.20
→ 融合后：更适合下一次测验

Day 3 14:00：距离上次学习 2 天
→ Retrieval 遗忘：0.35 × exp(-0.3 × 2) = 0.19
→ Storage 遗忘：0.20 × exp(-0.1 × 2) = 0.16

Day 3 14:00：做题“汉诺塔”，答对，用时 90 秒
→ Storage 大幅更新：0.16 → 0.65
→ 融合后：0.48

Day 10 10:00：距离上次学习 7 天
→ Retrieval 遗忘到较低水平
→ Storage 仍保留部分掌握状态
→ 调度 Agent 判断遗忘风险高，触发动作推荐
```

#### 6.1.10 与已有 DKT 改进模型的差异

| 模型 | 行为区分 | 遗忘机制 | 多模态 MOOC 行为 | 结构级行为建模 |
|---|---|---|---|---|
| DKT | 否 | 否 | 否 | 否 |
| DKT-F | 否 | 有，主要通过时间特征建模 | 否 | 否 |
| LFKT | 否 | 有，关注隐层衰减 | 否 | 否 |
| MFKT | 多特征拼接 | 有 | 部分支持 | 否 |
| Pierson MOOC + DKT | 面向 MOOC 行为 | 弱或无 | 支持 | 否 |
| SR-DKT | 长期储存 / 短期提取双状态建模 | 差异化遗忘速率 | 当前实验支持习题与视频；论坛为可扩展行为源 | 是 |

因此，SR-DKT 的创新不是简单把更多特征拼接进 DKT，而是在模型结构层面将长期储存和短期提取分开建模，并用差异化遗忘机制和双状态融合解释两类状态对后续作答的不同影响。

### 6.2 学习动作推荐模块

动作推荐 Agent 根据双状态结果和课程知识图谱生成学习动作。基本策略可以是：

1. 找出双状态都较低或遗忘风险较高的知识点；
2. 根据 `S_t` 与 `R_t` 的组合判定适合的动作类型；
3. 在课程知识图谱中定位这些知识点对应的视频、习题和讨论资源；
4. 根据知识点先修关系和资源难度映射到具体资源；
5. 输出下一步学习动作或一个短期动作序列。

示例：

```text
学生 A 当前状态：
- K3：S=0.42, R=0.38
- K5：S=0.78, R=0.31
- 遗忘风险：高

系统推荐：
1. K3：观看讲解视频 V12
2. K3：完成低难度练习 Q31-Q35
3. K5：安排间隔复习 + 简短复述任务
4. 48 小时后再次触发主动检索测验
```

### 6.3 反事实离线验证机制

为了避免“用自己的模型验证自己”的质疑，本文把验证拆成两层：

1. **主验证信号**：真实后续行为的 `HR@10` 和 `NDCG@10`；
2. **辅助解释信号**：SR-DKT 估计的双状态变化 `ΔKS`。

#### 6.3.1 主验证：真实后续行为命中率

系统用学生真实后续 30% 行为序列验证推荐结果，统计推荐动作是否映射到真实访问资源，从而得到 `HR@10` 和 `NDCG@10`。

#### 6.3.2 辅助验证：双状态变化

```text
ΔKS = f(S_after, R_after) - f(S_before, R_before)
```

其中 `f` 是加权融合函数，用于解释推荐动作为什么有效，但不作为唯一验证依据。

#### 6.3.3 纠正机制

若主验证信号和辅助解释信号都不理想，则调整动作类型、资源难度和复习间隔；若主验证命中但 `ΔKS` 仍偏低，则优先调整动作映射权重。

### 6.4 SM-2 调度与复习安排

SM-2 是 **SuperMemo Algorithm 2** 的缩写，由 Piotr Wozniak 于 1987 年提出，是经典的间隔重复算法。它的核心思想是：根据学习者对某个知识点的记忆质量，动态调整下一次复习的时间间隔。

在传统应用中，SM-2 常用于 Anki 等记忆卡片系统。例如，学生背单词后手动给出 0-5 分的记忆质量评分，系统根据评分决定下次复习是 1 天后、6 天后，还是更久之后。

本文不直接照搬传统 SM-2，而是将其嵌入调度 Agent 中，用于 MOOC 知识点复习安排。其关键改造是：传统 SM-2 依赖人工打分，本文可以用 SR-DKT 自动估计的双状态结果来替代或辅助记忆质量评分。

#### 6.4.1 SM-2 核心参数

| 参数 | 含义 | 初始值或说明 |
|---|---|---|
| `I(n)` | 第 `n` 次复习后的间隔天数 | `I(1)=1`，`I(2)=6` |
| `EF` | Ease Factor，易度系数 | 通常初始值为 2.5 |
| `q` | 记忆质量评分 | 0-5 分 |
| `n` | 某知识点的连续成功复习次数 | 从 0 或 1 开始计数 |

其中，`q` 的含义可以设计为：

| 分数 | 含义 | 处理方式 |
|---|---|---|
| 5 | 完美掌握，几乎不费力 | 正常增加复习间隔 |
| 4 | 正确掌握，稍有犹豫 | 正常增加复习间隔 |
| 3 | 勉强掌握，接近遗忘 | 增加间隔，但降低 `EF` |
| 2 | 掌握失败，但能识别答案 | 重置学习 |
| 1 | 明显不会 | 重置学习 |
| 0 | 完全没印象 | 重置学习 |

#### 6.4.2 SM-2 基本流程

SM-2 的计算流程可以概括为：

```text
如果 q < 3：
    n = 0
    下次复习间隔 = 1 天

如果 q >= 3：
    n = n + 1

    如果 n == 1：
        下次复习间隔 = 1 天
    如果 n == 2：
        下次复习间隔 = 6 天
    如果 n > 2：
        下次复习间隔 = 前次间隔 × EF
```

易度系数 `EF` 的更新公式为：

```text
EF' = EF + (0.1 - (5 - q) × (0.08 + (5 - q) × 0.02))
```

通常需要设置下界，避免 `EF` 过低：

```text
如果 EF' < 1.3，则 EF' = 1.3
```

在本文系统中，也可以设置上界，例如 `EF' <= 2.5`，使复习间隔不会扩张过快。

#### 6.4.3 在本文系统中的改造方式

传统 SM-2 与本文系统的差异如下：

| 维度 | 传统 SM-2 | 本文的 SM-2 嵌入 |
|---|---|---|
| 复习对象 | 单词、卡片、知识卡 | MOOC 知识点、视频片段、习题集合 |
| 记忆评估 | 学习者人工打分 | SR-DKT 自动预测双状态，也可结合测验表现 |
| 触发条件 | 到达复习日期 | 掌握度低于阈值或到达复习日期 |
| 更新依据 | 人工评分 `q` | 掌握度变化、测验正确率、资源完成情况 |
| 调度执行者 | 记忆软件 | 调度 Agent |

具体来说，可以将 SR-DKT 输出的双状态融合分数映射为 SM-2 的 `q` 分数：

| SR-DKT 融合分数 | 映射评分 `q` | 调度含义 |
|---|---|---|
| `p >= 0.90` | 5 | 掌握非常好，延长复习间隔 |
| `0.80 <= p < 0.90` | 4 | 掌握较好，正常推进 |
| `0.65 <= p < 0.80` | 3 | 勉强掌握，需要较快复习 |
| `0.50 <= p < 0.65` | 2 | 掌握不足，重置或安排补救 |
| `p < 0.50` | 1 | 明显薄弱，立即推荐补救资源 |

这样，系统就可以从“人工记忆评分”转向“模型自动评估”，实现更适合 MOOC 场景的复习调度。

#### 6.4.4 调度 Agent 的工作示例

以“递归函数”知识点为例：

```text
第 1 次学习：
SR-DKT 预测融合分数 = 0.60
映射 q = 2
处理：掌握不足，调度 Agent 推荐补救视频和基础习题
下次复习：1 天后

第 2 次学习后：
SR-DKT 预测融合分数 = 0.85
映射 q = 4
处理：掌握较好，进入正常复习序列
下次复习：6 天后

第 3 次复习后：
SR-DKT 预测融合分数 = 0.76
映射 q = 3
处理：勉强掌握，降低 EF，缩短后续间隔
下次复习：根据 I(n) × EF 计算

第 4 次复习前：
SR-DKT 预测融合分数下降到 0.58
映射 q = 2
处理：触发补救学习，重新安排复习
```

#### 6.4.5 本文中的调度规则

调度 Agent 可以综合 SR-DKT 和 SM-2，形成以下规则：

- 当某个知识点掌握度低于阈值时，立即推荐补救资源；
- 当某个知识点到达 SM-2 复习时间点时，推荐复习资源；
- 当某个知识点掌握度随时间明显下降时，提前触发复习；
- 优先复习掌握度低、遗忘风险高、且位于后续学习路径先修位置的知识点；
- 对掌握度高且稳定的知识点，逐步延长复习间隔，减少重复学习负担。

因此，SM-2 在本文中不是独立的背诵算法，而是调度 Agent 的核心机制之一。它使系统不仅能够推荐“下一个学习资源”，还能根据知识遗忘风险持续安排复习，从而形成长期学习支持。

## 10. 创新点梳理

本文的创新点可以归纳为三个层面：第一，提出 SR-DKT 双状态认知知识追踪模型，解决传统知识追踪只输出单一掌握度、无法区分长期储存与短期提取的问题；第二，提出基于双状态的学习动作推荐机制，将认知状态用于动作选择与资源映射；第三，提出双层可信验证闭环，将真实后续行为作为主验证信号、双状态变化作为辅助解释信号，解决现有多 Agent 学习路径规划缺少可信验证的问题。

### 10.1 创新点一：SR-DKT 双状态认知行为知识追踪模型

#### 10.1.1 核心问题

MOOC 学习场景里，最关键的问题不只是“学生会不会”，而是“学生现在能不能想起来”和“学生能不能长期记住”。传统知识追踪往往只给出单一掌握度，无法区分短期提取强度和长期储存强度，因此在长周期课程中很容易把“做对一次”误判成“真正掌握”。

#### 10.1.2 理论依据

SR-DKT 以 Bjork 的记忆双强度理论和测试效应为基础，将学习状态拆分为 Storage 与 Retrieval 两个维度。主动检索行为更有利于长期储存，被动接收行为更容易体现在短期提取上，因此二者应分别建模。

#### 10.1.3 模型结构

```text
MOOC 行为序列
     ↓ 分类
Storage 分支 → LSTM_S → Storage Head → S_t
Retrieval 分支 → LSTM_R → Retrieval Head → R_t
                    ↓
             双状态融合层
             P(correct)=σ(w1·S_t + w2·R_t - b)
```

其中 `λ_R > λ_S`，表示短期提取状态衰减更快，长期储存状态更稳定。

#### 10.1.4 与已有模型的差异

| 已有模型 | 主要特点 | 与本文差距 |
|---|---|---|
| DKT / DKT-F / LFKT | 单状态或单一遗忘建模 | 没有双状态解耦 |
| MFKT / BEKT | 行为特征融合 | 偏特征补丁，不是双状态认知建模 |
| KADKT / LB-KT | 行为导向建模 | 仍停留在行为特征层 |
| DDKT | 难度双通道 | 关注题目难度，不关注储存与提取 |
| 本文 SR-DKT | Storage / Retrieval 双状态 + 差异化遗忘 | 直接对应长期储存与短期提取 |

#### 10.1.5 验证方式

通过 SR-DKT 主实验和消融实验验证双状态建模是否优于单状态模型，指标包括 `AUC`、`ACC` 和 `F1`。

### 10.2 创新点二：基于双状态的学习动作推荐与离线验证机制

#### 10.2.1 核心问题

现有 MOOC 推荐系统大多直接推荐视频、习题或讨论资源，容易和传统资源推荐系统混在一起，缺少“应该做什么学习动作”的认知层表达。换句话说，系统往往是在推荐资源，而不是推荐策略。

#### 10.2.2 本文创新

本文提出基于双状态的学习动作推荐机制：先由 SR-DKT 输出 `S_t` 与 `R_t`，再根据双状态组合选择六类学习动作，并把动作映射到具体资源。

| 双状态组合 | 遗忘风险 | 推荐动作 | 资源类型 |
|---|---|---|---|
| `S` 低，`R` 低 | 低 | 观看讲解 | 视频资源 |
| `S` 低，`R` 中 | 低 | 低难度练习 | 基础习题 |
| `S` 低，`R` 高 | 低 | 主动检索测验 | 中等难度习题 |
| `S` 高，`R` 低 | 高 | 间隔复习 | SM-2 触发的复习资源 |
| `S` 高，`R` 低 | 低 | 解释 / 讨论 | 论坛发帖或概念复述 |
| `S` 高，`R` 高 | 低 | 进阶挑战 | 高难度习题或下一知识点 |

这使推荐输出从“资源列表”升级为“认知动作指令”，再由动作映射到资源池。

#### 10.2.3 反事实离线验证

本文在 Verify 层采用双层验证：

- 主验证信号：真实后续学习行为的 `HR@10`、`NDCG@10`；
- 辅助解释信号：SR-DKT 估计的 `ΔKS`。

这样可以避免“只用自己的模型状态解释自己的推荐效果”的自证问题。

#### 10.2.4 与已有系统的差异

| 维度 | MALPP | 本文系统 |
|---|---|---|
| 推荐单位 | 资源序列 | 学习动作 + 资源映射 |
| 知识状态 | 无显式双状态 | 有 SR-DKT 双状态 |
| 验证方式 | 路径合理性 | 真实行为命中 + `ΔKS` 辅助解释 |
| 迭代目标 | 路径规划优化 | 动作策略持续优化 |

### 10.3 两个创新点之间的关系

两个创新点形成“感知 → 决策 → 验证”的闭环。

```text
创新点一：SR-DKT
解决“学生当前能不能提取、长期能不能记住”
        ↓
创新点二：学习动作推荐
解决“当前更适合做什么动作”
        ↓
离线验证机制
解决“这个动作是否真的有效”
        ↓
形成完整闭环：双状态感知 → 动作推荐 → 真实验证 → 策略纠正
```

一句话概括就是：

- SR-DKT 负责“看清学生现在是存得住还是想得起”；
- 学习动作推荐负责“把双状态转成具体学习动作”；
- 离线验证负责“判断动作是否真正带来收益”。

### 10.4 盲审专家视角下的创新价值

| 创新点 | 专家看到的价值 | 对应盲审指标 |
|---|---|---|
| SR-DKT | 用记忆双强度理论把单一掌握度拆成双状态，避免只做特征拼接 | 创新性、学术性 |
| 学习动作推荐 | 从资源推荐提升为认知动作推荐，更贴近学习策略 | 应用性、表达力 |
| 反事实离线验证 | 用真实后续行为验证推荐效果，避免自证有效 | 严谨性、可验证性 |
| 两者融合 | 形成“感知-决策-验证”完整链条 | 完整性、逻辑性 |

总体来看，这套新方案比原来的“行为感知 + 多 Agent”更聚焦，且更适合答辩时讲成一条清晰主线。

## 11. 与已有研究的差异

### 11.1 与田雪涛博士论文的差异

相关研究：

**田雪涛：《面向 MOOC 的个性化学习路径推荐关键问题研究》，北京交通大学，2022。**

| 维度 | 田雪涛研究 | 本文研究 |
|---|---|---|
| 核心方法 | 用户画像、知识概念图谱、课程序列推荐 | SR-DKT、学习动作推荐、SM-2 调度 |
| 能力追踪 | 偏课程级或主题级 | 知识点级掌握度追踪 |
| 推荐方式 | 传统推荐系统思路 | 动态知识状态驱动推荐 |
| 动态优化 | 静态推荐为主 | 基于 Verify→Correct 的 Harness 迭代优化 |
| 系统形态 | 推荐模型 | 知识追踪驱动的教育 Harness 系统 |

结论：田雪涛的研究是最接近的传统推荐系统方向，但本文可以在其基础上进行 Agent 化升级，形成差异化创新。

### 11.2 与 MAIC / OpenMAIC 的差异

| 维度 | MAIC / OpenMAIC | 本文研究 |
|---|---|---|
| 主要目标 | LLM 驱动的 AI 课堂与教学互动 | MOOC 个性化学习路径推荐 |
| 应用场景 | 课程内容生成、课堂互动 | 学习行为分析、路径规划 |
| 知识追踪 | 通常没有 DKT 机制 | 引入 SR-DKT |
| 推荐机制 | 非核心重点 | 本文核心任务 |
| 动态调度 | 较少涉及 | 引入 KT-Harness 和 SM-2 复习调度 |

结论：MAIC / OpenMAIC 虽然也是多智能体教育系统，但主要面向课堂教学体验；本文面向学习路径推荐，技术重点不同。

### 11.3 与 DDKT 的差异

相关研究：

**DDKT: LLM-driven Effective Knowledge Tracing by Integrating Dual-channel Difficulty, arXiv 2025.**

DDKT 与本文最容易发生“名称层面”的混淆，因为它使用了 Dual-channel Knowledge Tracing 的表述。但二者的双路径含义、任务场景和技术路线并不相同。

| 维度 | DDKT | 本文 SR-DKT |
|---|---|---|
| 核心概念 | Dual-channel Difficulty，主观难度与客观难度 | Behavior-Aware，主动检索与被动接收行为 |
| 关注问题 | 题目难不难、难度如何影响作答 | 学生怎么学、行为类型如何影响掌握与遗忘 |
| 技术架构 | Transformer + LLM + RAG | LSTM 双路径 + 差异化遗忘 + 注意力融合 |
| 遗忘机制 | 非核心 | 核心组件，区分 `λ_a` 与 `λ_p` |
| 数据场景 | XES3G5M、Eedi 等习题平台 | MOOCCubeX，当前核心实验使用习题作答与视频观看行为 |
| 行为区分 | 不区分主动 / 被动学习行为 | 显式区分主动检索与被动接收 |

结论：DDKT 与本文“名称曾经相近”，但研究对象不同。本文已将方法命名调整为 SR-DKT，强调 Storage / Retrieval 双状态而非难度双通道，从根源上降低命名撞车风险。

### 11.4 与 KADKT 的差异

相关研究：

**KADKT（Knowledge Absorptive Capacity Deep Knowledge Tracing）, 华南师范大学学报, 2025.**

KADKT 是本文需要认真对比的一篇工作。它从“知识吸收能力”的视角，将行为划分为显性行为与隐性行为，并在知识追踪中引入遗忘模块。

| 维度 | KADKT | 本文 SR-DKT |
|---|---|---|
| 行为分类 | 显性行为 vs 隐性行为 | 主动检索 vs 被动接收 |
| 理论视角 | 知识吸收能力的工程视角 | 测试效应与主动学习的认知科学视角 |
| 模型结构 | 特征级融合为主 | 双路径 LSTM 结构级建模 |
| 遗忘机制 | 有遗忘模块 | `λ_p > λ_a` 差异化遗忘约束 |
| 数据场景 | ASSISTments、Junyi 等习题平台 | MOOCCubeX，多模态 MOOC 场景 |
| 贡献重点 | 显隐行为与吸收能力 | 认知机制分路径 + MOOC 场景迁移 |

结论：KADKT 说明“显性/隐性行为”是一个可行的 KT 改进方向，但它仍然属于工程化行为划分，和本文的主动检索 / 被动接收认知分路径不同。本文的优势在于把测试效应作为理论锚点，并把该理论落实为结构级双路径与差异化遗忘约束。

### 11.5 与 BEKT 的差异

相关研究：

**BEKT（Behavior-Enhanced Knowledge Tracing）, 2026.**

BEKT 的真实结构更接近“在 AKT 骨架上叠加行为特征补丁”的改良模型。它通过互信息筛选出少量与习题作答相关的行为特征，再用 Cross-Feature 和 MLP 将这些特征注入知识状态。

| 维度 | BEKT | 本文 SR-DKT |
|---|---|---|
| 行为来源 | 习题平台中的少量行为特征 | MOOC 行为，当前核心实验使用习题作答与视频观看 |
| 行为分类依据 | 数据驱动特征筛选 | 认知科学测试效应下的主动检索 / 被动接收 |
| 模型结构 | AKT 骨架 + Cross-Feature + MLP | 双路径 LSTM + 注意力融合 |
| 遗忘机制 | 无显式遗忘模块 | `λ_p > λ_a` 差异化遗忘 |
| 场景适配 | ASSISTments / 习题平台 | MOOCCubeX / MOOC 平台 |
| 贡献重点 | 行为特征增强 | 认知机制分路径 + MOOC 多模态迁移 |

结论：BEKT 证明“行为特征可以增强 KT”，但它本质上仍是特征级补丁式融合，没有独立的双状态认知路径，也没有遗忘机制，更无法迁移到 MOOC 长周期多模态学习场景。因此，SR-DKT 在理论依据、结构设计和场景适配上都不同于 BEKT。

### 11.6 与 LB-KT 的差异

相关研究：

**Learning Behavior-oriented Knowledge Tracing (LB-KT), KDD 2023.**

LB-KT 是目前与 SR-DKT 最接近的行为导向知识追踪工作。它已经注意到“学习行为类型”对知识追踪的重要性，但其核心仍停留在行为特征级别的建模。

| 维度 | LB-KT | 本文 SR-DKT |
|---|---|---|
| 关注对象 | 行为类型作为输入特征 | 主动检索 / 被动接收的认知机制分路径 |
| 理论依据 | 主要是行为导向建模 | 测试效应 + 主动学习理论 |
| 模型结构 | 单路径或特征融合式 KT | 双路径 LSTM + 注意力融合 |
| 遗忘机制 | 无差异化遗忘约束 | `λ_p > λ_a` 差异化遗忘 |
| MOOC 场景 | 不一定针对 MOOC 多模态行为 | 面向 MOOCCubeX 习题作答与视频观看的真实 MOOC 场景 |
| 贡献重点 | 行为作为知识追踪输入 | 行为认知机制 + 遗忘约束 + 结构级建模 |

结论：LB-KT 证明“学习行为”值得被纳入知识追踪，但它并没有上升到“双状态认知”的结构级建模，也没有差异化遗忘和 Harness 式系统联动。因此，SR-DKT 仍然具有明确的结构级和理论级差异。

### 11.7 与 MALPP 的差异

相关研究：

**MALPP: Multi-Agent Learning Path Planning via LLMs, arXiv 2026.**

MALPP 已经包含 Reflection Agent，因此本文不再把“设置 Reflection Agent”作为独立创新点。本文与 MALPP 的关键差异在于：MALPP 主要依赖 LLM 的语义推理进行学习路径规划，而本文将 SR-DKT 输出的双状态向量作为 KT-Harness 的动态约束、验证标准和纠正信号。

| 维度 | MALPP | 本文系统 |
|---|---|---|
| 推荐依据 | LLM 对学习目标、资源和路径的语义推理 | SR-DKT 双状态向量 + 课程知识图谱 |
| 知识追踪 | 无显式 DKT / KT 模型 | 有 SR-DKT 细粒度双状态追踪 |
| 遗忘建模 | 通常无 | SR-DKT 差异化遗忘 + SM-2 调度 |
| Reflection / Correct 目标 | 评估路径规划是否合理 | 评估 `ΔKS` 是否达标，并纠正推荐策略 |
| 个性化依据 | 学习目标和文本上下文 | 学生历史行为、知识状态、遗忘风险、资源图谱 |
| 系统范式 | 多 Agent 路径规划 | KT-Harness 教育驾驭系统 |

结论：MALPP 已覆盖“多 Agent + Reflection 路径规划”的一部分思路，因此本文的创新重心应放在 **学习动作推荐 + 反事实验证**，即用 SR-DKT 双状态向量动态驱动 Constrain、Verify 和 Correct，而不是单独强调 Reflection Agent。

### 11.8 与 MemoryKT 等记忆过程类 KT 的差异（最近邻，必须澄清）

相关研究：

- **MemoryKT: An Integrative Memory-and-Forgetting Method for KT, arXiv 2508.08122, 2025.**
- **CPF: Concept-driven Personalized Forgetting KT, arXiv 2404.12127, 2024.**
- **GRKT: Graph-based Reasonable KT（教育学理论驱动）, arXiv 2406.12896, 2024.**
- **FoLiBi: Forgetting-aware Linear Bias for Attentive KT, arXiv 2309.14796, 2023.**

其中 **MemoryKT（2025）是与本文理论内核最接近的工作**：它同样以"存储/提取"和遗忘为核心，显式建模 encoding–storage–retrieval 三阶段记忆过程（用时序 VAE：编码器=encoding、LSTM=storage、解码器=retrieval），并设计**个性化遗忘模块**，从每个学生历史行为统计出"个性化遗忘分数"调节存储强度。因此"存储-提取 + 差异化遗忘"本身**已不再是独占的创新点**，本文必须明确与之划界。

| # | 维度 | MemoryKT (2025) | 本文 SR-DKT | 本文是否独有 |
|---|---|---|---|:---:|
| 1 | "存储/提取"含义 | 记忆**过程阶段**（编码→存储→提取流水线，Atkinson-Shiffrin 式） | Bjork 双强度理论的两个**强度状态**（存储强度 vs 提取强度）**并行建模** | 理论谱系不同 |
| 2 | 结构 | 单流 VAE + LSTM | **双路径**：Storage 走做题、Retrieval 走视频，参数不共享 | ✅ |
| 3 | 遗忘差异化的"轴" | **跨学生**个性化（per-student 遗忘分） | **跨记忆类型**（`λ_R > λ_S`，提取比存储忘得快） | ✅ 不同轴 |
| 4 | 数据 / 模态 | 纯做题数据集（ASSIST09/15 等） | **MOOC 多模态行为**（`watch_ratio` / `is_replay` 视频信号） | ✅ |
| 5 | 应用 | 纯 KT 预测 | KT + **学习动作推荐与 KT-Harness 闭环** | ✅ |

**诚实的短板（须在论文中正面处理）**：MemoryKT 的核心卖点是**跨学生个性化遗忘**，而本文 `λ_S`、`λ_R`、`β` 当前为**全局参数**（所有学生共用），在"个性化"这一维度弱于 MemoryKT。应对方式有二：（a）将本文创新口径明确钉在"**按学习行为类型分路 + 跨记忆类型差异化遗忘 + 推荐应用**"，主动声明个性化遗忘非本文主张；（b）后续为 `λ / β` 增加由学生历史统计驱动的轻量 per-student 调节，补齐个性化维度。

**创新口径结论**：在 MemoryKT/CPF 之后，本文**不再以"提出存储-提取双状态遗忘"为头条创新**，而应表述为——"基于 Bjork 双强度理论，**首次将存储/提取强度与主动做题/被动观看两类学习行为分路绑定**，配合跨记忆类型差异化遗忘（`λ_R > λ_S`），并由双状态驱动 MOOC 学习动作推荐（KT-Harness）"。GRKT（理论驱动 + 可解释 + 赢基线）是本文叙事范本；FoLiBi 表明此类遗忘改进的现实增益约 +2.58% AUC，提示本文不应以"大幅领先准确率"为主要卖点，而以**可解释 + 行为分路 + 推荐应用**立论。

### 11.9 研究空白总结

| 已有研究 | 主要空白 | 本文机会 |
|---|---|---|
| 传统 MOOC 推荐系统 | 缺少多智能体协作与动态验证纠正 | KT-Harness 教育驾驭框架 |
| KADKT 等显隐行为 KT | 关注工程视角的显性 / 隐性划分 | SR-DKT 双状态追踪 |
| DDKT 等双通道 KT | 关注难度通道，不关注 MOOC 行为类型 | SR-DKT 双状态追踪 |
| BEKT 等行为增强 KT | 行为作为特征补丁，缺少结构级路径与遗忘 | SR-DKT 结构级双状态与差异化遗忘 |
| DKT-F / LFKT 等遗忘 KT | 多集中在习题场景 | MOOC 主动 / 被动行为差异化遗忘 |
| MemoryKT / CPF 等记忆-遗忘 KT（2024-2025） | 记忆过程/个性化遗忘，单流、纯做题、无推荐 | 行为分路（做题/视频）+ 跨记忆类型遗忘 + 推荐应用 |
| MALPP 等多 Agent 路径规划 | 有 Reflection，但缺少知识追踪和 Harness 动态约束 | KT-Harness 教育驾驭系统 |
| 软件工程 Harness | 有 Constrain / Verify / Correct，但约束多为代码规则 | 用 SR-DKT 双状态生成教育场景动态约束 |
| 静态学习路径推荐 | 缺少基于掌握度变化的反馈优化 | KT-Harness 迭代收敛闭环 |

## 15. 实验算力需求与可行性

本研究的实验算力需求整体较低，主要计算压力集中在 SR-DKT 模型训练和多组消融实验。学习动作推荐系统运行、SM-2 调度计算和统计分析都不需要高性能 GPU，因此个人电脑、学校服务器或普通云 GPU 平台均可完成。整体上，使用 RTX 3060 12GB 或百度 AI Studio 免费 GPU 即可完成实验，全部核心实验预计约 10-15 小时。

### 15.1 主要计算任务

| 模块 | 计算任务 | 复杂度 |
|---|---|---|
| SR-DKT 模型训练 | 双状态 LSTM 神经网络训练 | 中等 |
| 学习动作推荐系统运行 | 规则推理 + 流程编排 | 低到中等 |
| SM-2 调度计算 | 间隔重复公式计算 | 极低 |
| 消融实验 | 多组模型训练和对比 | 中等 × 多次 |
| 数据分析 | 统计检验、指标计算、可视化 | 低 |

核心算力需求可以概括为：

| 资源 | 需求 |
|---|---|
| GPU | RTX 3060 12GB 即可 |
| 训练时间 | 约 10-15 小时，取决于课程数量、样本规模和重复实验次数 |
| 推荐平台 | 百度 AI Studio，MOOCCubeX 已上线，适合快速复现实验 |

其中，真正需要 GPU 的主要是 SR-DKT 模型训练。学习动作推荐与验证系统如果调用 OpenAI、DeepSeek、通义千问等 API，本地只需要负责逻辑编排，不需要本地部署大模型。

### 15.2 SR-DKT 模型规模估算

SR-DKT 本质上是一个以双状态双路径 LSTM 为核心的小型序列模型，参数量远小于大语言模型。

| 组件 | 典型规模 | 说明 |
|---|---|---|
| 输入维度 | 100-500 维 | 视频、习题、行为类型、时间特征等嵌入 |
| LSTM 隐层 | 128-256 单元 | 标准 DKT 配置 |
| 输出维度 | 100-500 维 | 知识点掌握概率或相关预测目标 |
| 参数量 | 约 50 万-200 万 | 属于小模型 |

显存可以粗略估算为：

```text
模型参数：
200 万参数 × 4 字节 × 3（参数 + 梯度 + 优化器状态）
≈ 24 MB

批次激活值：
64 条序列 × 200 步 × 256 维 × 4 字节
≈ 13 MB

加上缓存和框架开销：
总显存通常在 50-100 MB 量级，实际训练预留 2-4 GB 显存已经足够。
```

因此，SR-DKT 不需要 A100 或大规模 GPU 集群，普通消费级显卡即可完成训练。

| LLM 调用 | API 调用 | 无需本地 GPU |
| LangGraph / AutoGen | CPU 即可 | 负责 Agent 流程编排 |
| 知识图谱查询 | CPU + 内存 | 通常为毫秒级或秒级 |
| 推荐规则计算 | CPU 即可 | 计算量较低 |

#### SM-2 调度计算

SM-2 只包含简单的加法、乘法和阈值判断，单次更新是 `O(1)` 常数复杂度。即使对 10 万个知识点进行调度计算，计算量也可以忽略不计。

```text
每个知识点：
读取掌握度 → 映射 q → 更新 EF → 计算下次复习间隔

复杂度：
O(1)
```

### 15.8 算力结论

| 问题 | 结论 |
|---|---|
| 需要多少显存？ | 2-4 GB 基本够用，推荐 12GB 更稳 |
| 需要多强 GPU？ | RTX 3060 级别即可，无需 A100 |
| 能否只用 CPU？ | 小规模调试可以，正式训练建议使用 GPU |
| KT-Harness 是否需要本地大模型？ | 不需要，可调用 API |
| 云服务器成本高吗？ | 不高，AutoDL 或 AI Studio 可低成本完成 |
| 核心训练时长 | 约 10-15 小时 |

因此，本研究的算力门槛较低。SR-DKT 属于小模型，SM-2 与学习动作推荐编排计算量也不高，个人电脑、学校服务器或免费 / 低价云平台都可以完成全部实验。

## 17. 最终结论

“基于 SR-DKT 双状态与学习动作推荐的 MOOC 个性化学习路径推荐系统”是一个稳妥的硕士论文选题。

它的优势主要体现在：

1. 数据可获得：公开 MOOC 数据集规模大，且可直接用于离线实验；
2. 指标可量化：AUC、F1、HR@10、NDCG@10、完成率、掌握度提升等指标客观；
3. 方法有深度：SR-DKT 提供模型基础，学习动作推荐提供系统框架创新；
4. 实验可落地：可以完成 SR-DKT 主实验、SR-DKT 消融、动作推荐对比和反事实收敛实验；
5. 创新可解释：不是简单堆 Agent，也不是重复 MALPP 的 Reflection Agent，而是围绕 SR-DKT 双状态构建学习动作闭环。

最终可以将论文贡献概括为：

1. 提出一种面向 MOOC 场景的 SR-DKT 双状态知识追踪模型；
2. 设计一种基于双状态的学习动作推荐机制，将认知状态用于动作选择与资源映射；
3. 引入基于真实后续行为和 `ΔKS` 的双层验证机制与 SM-2 复习调度，实现推荐策略的迭代优化；
4. 在公开 MOOC 数据集上通过 SR-DKT 预测实验、动作推荐对比实验和反事实收敛实验验证系统有效性。

#### 训练集采样策略

为了在合理算力下完成多组对比实验，本文采用 **训练集分层采样 + 完整评估集** 的数据策略。出于进一步压缩算力成本的考虑，最终采用 **10% 训练子集**（早期曾使用 20% 子集，相关结果见第 20 节并已标注取代）：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 训练集 | 全量训练集的 10% | 从全量（96 万学生）分层采样 |
| 验证集 | 完整 | 全量（未采样） |
| 测试集 | 完整 | 全量（未采样） |
| 采样方法 | 分层采样 | 按序列长度分位数分 10 层，逐层等比例采样 |
| 采样种子 | 42 | 保证可复现 |
| 采样脚本 | `subsample_mooc.py` | 提交进仓库，固定种子，仅采样 train |
| 数据目录 | `data/mooc_10pct` | val/test/meta/encoder 硬链接复用全量 |

**为什么从全量直接采 10%、而非嵌套在 20% 子集内：** 随机样本的随机子样本仍是对全量人群的无偏样本，二者代表性相同；从全量直接采更简单，且不依赖此前手动切分的 20% 学生名单。仅当需要做"10% vs 20% 数据量"的配对消融时，嵌套设计才有降噪意义，本文不做该对比。

采用 10% 训练子集 + 全量验证/测试评估的方式，既进一步控制了实验算力成本，又保证了评估结果的统计可靠性（全量测试集覆盖所有学生）。**关键对齐要求：** 所有基线模型（DKT、DKVMN、SAKT、AKT 等）必须在**同一个 10% 训练子集 + 同一评估集**上重新训练与评估，第 20 节中基于 20% 子集得到的旧结果不可与 10% 结果直接比较。

### 18.3 基线模型对齐说明

本文采用“双轨基线”方案：标准 KT 基线优先使用 pyKT 统一框架复现，自有代码用于 SR-DKT、消融实验和行为增强补充基线。

| 轨道 | 用途 | 模型 | 数据字段 | 说明 |
|------|------|------|---------|------|
| pyKT 标准基线 | 减少自实现偏差 | DKT、DKT+、DKVMN、SAKT、AKT、simpleKT | `question_id`、`skill_id`、`correct` | 使用 pyKT 统一训练和评估协议 |
| pyKT 时间 / 遗忘基线 | 对比遗忘建模 | DKT-Forget、LPKT、HawkesKT、FoLiBiKT / LefoKT | `question_id`、`skill_id`、`correct`、`timestamp`、`duration` / `delta_t` | 尽量保留真实时间字段；若 `duration` 缺失，则标注为 `NA` |
| 自有行为增强基线 | 对比行为特征融合 | LBKT、BEKT、LBKT+video、BEKT+video | `attempt`、`hint`、`delta_t`、可选 `watch_ratio` / `is_replay` | 用于验证 SR-DKT 不是单纯依赖额外视频特征 |
| 自有创新模型 | 验证本文方法 | SR-DKT、SR-DKT w/o video、SR-DKT 各消融 | `kc_id`、`correct`、`delta_t`、`attempt`、`watch_ratio`、`is_replay` | 保留 Storage / Retrieval 双状态、差异化遗忘和融合机制 |

当前仓库中的 `baselines/run_baselines.py` 实现了 DKT、DKT-F、SAKT、AKT、DKVMN、LBKT、BEKT 等模型，可作为统一输入设置下的补充对比。但为了避免“自实现基线可能偏弱”的质疑，正式论文结果建议以 pyKT 跑出的标准基线作为主对照，并在附录或补充实验中报告本仓库自实现基线结果。

在当前阶段，实验执行优先级确定如下。第一批模型用于形成论文主实验的最小闭环，当前阶段主要跑第一批；第二批模型用于增强学习过程、时间建模和行为公平性论证。

| 执行批次 | 模型 | 放入哪张表 | 作用 |
|--------|------|------------|------|
| 第一批必跑 | pyKT: DKT、DKVMN、SAKT、AKT、simpleKT、DKT-Forget；本文模型: SR-DKT | 标准 KT 对比表、时间 / 遗忘对比表 | 覆盖经典 KT、记忆网络、注意力 KT、强标准 KT 与遗忘 KT，支撑 SR-DKT 主效果 |
| 第二批建议跑 | LPKT、HawkesKT、LBKT、BEKT、SR-DKT w/o video | 时间 / 遗忘对比表、行为增强公平性表、消融 / 公平性表 | 增强学习过程建模、时间交叉效应和视频特征公平性论证 |

其中，第一批外部 baseline 全部优先使用 pyKT 统一训练和评估协议；本仓库自实现的 `DKT / DKVMN / SAKT / AKT` 仅作为预实验、调试或附录补充结果。`LPKT` 与 `HawkesKT` 是第二批 pyKT 扩展模型。

字段对齐原则如下：

| 字段 | pyKT 标准基线 | pyKT 时间 / 遗忘基线 | 自有行为基线 | SR-DKT |
|------|---------------|----------------------|--------------|--------|
| `question_id` | 使用 | 使用 | 可选 | 当前不直接使用 |
| `skill_id` / `kc_id` | 使用 | 使用 | 使用 | 使用 |
| `correct` | 使用 | 使用 | 使用 | 使用 |
| `timestamp` | 可选 | 使用 | 可选 | 用于预处理阶段计算 `delta_t` |
| `duration` / `answer_time` | 通常不用 | LPKT 等模型需要 | 可选 | 当前不使用 |
| `delta_t` | 通常不用 | 可由 timestamp 计算 | 使用 | 使用 |
| `attempt` | 通常不用 | 通常不用 | 使用 | 使用 |
| `hint` | 通常不用 | 通常不用 | 可使用；MOOCCubeX 当前为 0 | 使用；MOOCCubeX 当前为 0 |
| `watch_ratio` | 不使用 | 不使用 | 增强版可使用 | 使用 |
| `is_replay` | 不使用 | 不使用 | 增强版可使用 | 使用 |

`question_id` 与 `timestamp` 不应在预处理阶段丢弃。`question_id` 用于题目级评估和 AKT 等题目敏感模型；`timestamp` 用于 DKT-Forget、LPKT、HawkesKT 等时间 / 遗忘模型。SR-DKT 本身主要使用 `kc_id` 和相邻交互时间间隔 `delta_t`，但保留原始 `question_id` 与 `timestamp` 可以让 pyKT 基线更完整，也使论文的数据处理更可复现。

## 19. MOOCCubeX 数据集与预处理详解

### 19.1 数据来源与原始规模

MOOCCubeX 由清华大学知识工程实验室（THU-KEG）发布，基于学堂在线平台的真实 MOOC 学习行为数据。原始数据为 JSON 行格式，每个文件一行一个 JSON 对象。

| 数据类型 | 文件 | 大小 | 记录数 | 说明 |
|---------|------|------|--------|------|
| 用户做题记录 | `relations/user-problem.json` | 22.5 GB | ~1.33 亿条 | 每行一个学生的一道题作答 |
| 用户视频观看 | `relations/user-video.json` | 3.2 GB | ~31 万条 | 每行一个学生对一个视频的观看片段 |
| 问题元数据 | `entities/problem.json` | 1.3 GB | — | 包含 `problem_id` → `exercise_id` 映射 |
| 视频信息 | `entities/video.json` | 608 MB | 230,263 个 | 视频总时长、片段标识 |
| 问题-练习映射 | `relations/exercise-problem.txt` | — | ~99.95% 覆盖率 | Tab 分隔，`Ex_XXXXX\tPm_XXXXX` |
| 视频 ID 桥接 | `relations/video_id-ccid.txt` | — | — | `V_XXXXX` ↔ hex ccid 映射 |

### 19.2 预处理流水线（6 步）

预处理脚本 `preprocess_mooc.py` 将上述 44GB 原始 JSON 转换为 6GB 的 pickle 训练文件：

```
Step 1: 加载 problem_id → exercise_id 映射
   ├─ 主源: relations/exercise-problem.txt (覆盖率 ~99.95%)
   └─ 补充: entities/problem.json (补充缺失)

Step 1.5: 加载 video_id → ccid 映射
   └─ V_XXXXX ↔ hex ccid: 桥接两种 ID 体系

Step 2: 加载视频时长索引
   └─ entities/video.json → {ccid: total_duration}

Step 3: 处理用户视频数据
   ├─ 读取 relations/user-video.json
   ├─ 计算 watch_ratio = 观看时长 / 视频总时长
   ├─ 检测 is_replay: 多片段按 start_point 排序后判断重叠
   └─ 输出: {user_id: [(video_id, watch_ratio, is_replay, last_epoch)]}

Step 4: 流式加载做题数据
   ├─ 流式读取 relations/user-problem.json (chunk_size=5M)
   ├─ 解析: problem_id → exercise_id, is_correct, attempts, submit_time
   ├─ 用 numpy 结构化数组存储: dtype=[pid, epoch, correct, attempt]
   └─ 24GB 内存 → 在 32GB 机器上可运行

Step 4.5: Top-K KC 过滤
   ├─ 按出现频率排序，保留 top-N exercise_id
   ├─ 其余映射为 "OTHER" 类
   └─ --max-kc 30000 时，OTHER 占比 ~15.3%

Step 5: 构建序列
   ├─ 按时间戳排序 + 计算 delta_t (天)
   ├─ 二分查找匹配视频特征 (epoch ≤ 做题时间)
   ├─ 输出 SR-DKT 7 元组: (kc_id, correct, delta_t, hint, attempt, watch_ratio, is_replay)
   └─ 就地编码: exercise_id → LabelEncoder → kc_id 整数

Step 6: 划分 & 保存
   ├─ 过滤 min_interactions ≥ 3 的学生
   ├─ 80/10/10 划分 train/val/test
   └─ 输出: train.pkl / val.pkl / test.pkl + meta_mooc.json + skill_encoder_mooc.pkl
```

### 19.3 内存优化措施

考虑到 44GB 原始数据在 32GB 内存环境下处理，预处理采用了多项优化：

| 优化 | 说明 | 效果 |
|------|------|------|
| 流式读取 | user-problem.json 逐行 JSON 解析，不一次性加载 | 避免 22.5GB 全量加载 |
| numpy 结构化数组 | 交互记录用 `np.dtype([(pid,i64),(epoch,f64),(correct,i1),(attempt,i2)])` | 单条记录 22B，比 Python dict 省 5-10x 内存 |
| 就地编码 | Step 6 直接修改 tuple 字段，不创建副本 | 避免 2x 内存峰值 |
| 及时释放 | 每个 Step 完成后 `del` 中间变量 | 控制峰值在 ~22GB |
| 字节级解析 | exercise-problem.txt 用 `rb` 模式 + 逐字节找 Tab | 避免 UTF-8 解码错误 |

总耗时约 64 分钟，内存峰值约 22GB。

### 19.4 输出序列格式

预处理后的 `train/val/test.pkl` 为 Python dict：

```python
{
    "user_0001": [
        (kc_id, correct, delta_t, hint, attempt, watch_ratio, is_replay),
        (kc_id, correct, delta_t, hint, attempt, watch_ratio, is_replay),
        ...
    ],
    "user_0002": [...],
    ...
}
```

| 字段 | 类型 | 值域 | 含义 |
|------|------|------|------|
| `kc_id` | int | 0 ～ 30,000 | LabelEncoder 编码后的知识点 ID（OTHER=30000） |
| `correct` | int | 0 / 1 | 该题是否答对 |
| `delta_t` | float | ≥ 0 | 距上一次交互的天数 |
| `hint` | float | 0 | MOOC 无 hint 信息，恒为 0 |
| `attempt` | float | ≥ 1 | 尝试次数 |
| `watch_ratio` | float | [0, 1] | 观看比例 = 观看时长 / 视频总时长 |
| `is_replay` | int | 0 / 1 | 是否存在回放片段 |

pyKT baseline 不复用 SR-DKT 的 7 元组 pkl，而是由独立脚本 `preprocess_mooc_pykt.py` 从原始 MOOCCubeX 文件重新导出标准 KT 交互 CSV：

```text
split,user_id,order_id,question_id,concept_id,response,timestamp,duration
```

两条预处理线的字段关系如下：

| 字段 | 保留原因 | 主要使用方 |
|------|----------|------------|
| `question_id` | 区分具体题目，支持题目级评估和题目敏感模型 | pyKT、AKT、question-level evaluation |
| `concept_id` / `kc_id` | 表示知识点 / 概念，是知识追踪的核心状态索引 | 全部 KT 模型、SR-DKT |
| `timestamp` | 保留真实提交时间，支持时间间隔计算与时间模型复现 | DKT-Forget、LPKT、HawkesKT |
| `delta_t` | 直接表征相邻学习行为间隔，是遗忘衰减输入 | SR-DKT、自有 DKT-F / BEKT |
| `duration` | 表征作答耗时，LPKT 等过程感知模型需要 | pyKT LPKT 等 |
| `attempt` | 表征尝试次数，反映学习过程行为强度 | SR-DKT、自有行为增强基线 |
| `hint` | 保留接口一致性；MOOCCubeX 当前无有效 hint 信号 | SR-DKT、自有行为增强基线 |
| `watch_ratio` | 表征被动视频接收强度 | SR-DKT Retrieval 分支、video-enhanced baseline |
| `is_replay` | 表征回看行为，反映重复接收 / 复习信号 | SR-DKT Retrieval 分支、video-enhanced baseline |

缺少 `question_id` 或 `timestamp` 不会使 SR-DKT 主模型失效，因为 SR-DKT 当前在 KC 层面建模，并直接使用 `delta_t` 进行遗忘衰减。但 pyKT 标准基线、题目级评估和时间 / 遗忘 baseline 需要这些字段，因此本文采用独立 pyKT 预处理脚本保留 `question_id`、`concept_id`、`timestamp` 和 `duration`。

### 19.5 视频特征提取详解

这是 SR-DKT Retrieval 分支的核心输入。提取流程：

1. **观看比例**（`watch_ratio`）：对每段视频的所有 `segment` 累加 `end_point - start_point`，除以视频总时长，截断到 [0, 1]。
2. **回放检测**（`is_replay`）：将所有 segment 按 `start_point` 排序，若存在相邻片段 `curr_start < prev_end`，则判定为回放。
3. **时间匹配**：对每个学生的做题记录按 epoch 排序，对每次做题使用二分查找定位该时间戳之前最近一次视频观看，将其 `watch_ratio` 和 `is_replay` 注入对应的序列位置。

最终视频特征覆盖率约 **9.5%**（即约 9.5% 的交互带有 watch_ratio > 0 或 is_replay > 0）。这部分数据是 SR-DKT 验证"被动行为影响 Retrieval 状态"信号的关键。

### 20.3 SR-DKT 与 DKT 性能对比

| 模型 | 最佳 Val AUC | 最佳 Val ACC | 收敛 epoch | 参数量级 | 差异（相对 DKT） |
|---|---:|---:|---:|---|---|
| **SR-DKT** | **0.8846** | **0.8800** | 9 | ~3.8M | **+0.0039**（+0.44%） |
| DKT | 0.8807 | 待补 | 10-11 | ~3.8M | 基准 |

**对比分析**：

1. **AUC 提升 0.44%**：在 MOOCCubeX 这种大规模 MOOC 场景下，0.4% 的 AUC 提升属于显著改进（已超过多数 KT 模型间的常规改进幅度）
2. **收敛更快**：SR-DKT 在第 9 轮达到最佳，DKT 在第 10-11 轮达到最佳
3. **过拟合更晚**：SR-DKT 在 11 轮才出现拐点，DKT 也在 11 轮拐点，但 SR-DKT 在拐点后下降更平缓（11→17 共下降 0.0035；DKT 11→18 下降 0.0024，量级接近）
4. **机制有效性**：SR-DKT 引入的差异化遗忘（λ_s/λ_r）+ 注意力融合机制在 7 元组输入（kc_id, correct, delta_t, hint, attempt, watch_ratio, is_replay）下表现优于单路径 DKT
